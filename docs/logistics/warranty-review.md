# 保固審核

**位置**：物流作業 → 保固審核 Tab

此 Tab 列出所有「保固狀態無法自動判定」或「需要人工審核」的 RMA 工單，由管理員手動確認保固判斷。

## 什麼時候需要操作這個 Tab？

- 系統顯示保固狀態為「⚠️ 無法判定」
- 序號無法解碼（格式異常、型號不符）
- 需要根據購買憑證覆蓋自動判斷結果

## 列表說明

顯示所有保固狀態待審核的工單，依申請時間排序。

每列顯示：
- RMA 號碼
- 客戶名稱
- 產品型號 / 序號
- 系統自動判定結果（無法判定 / 有疑問）
- 客戶提供的購買日期
- 申請天數

## 保固審核流程

### 步驟一：查看序號解碼資訊

在工單詳情頁「保固資訊」區塊查看：
- 序號解碼結果（成功 / 失敗）
- 自動判定的生產日期
- 系統建議的保固批次

### 步驟二：確認客戶資料

1. 確認客戶填寫的序號是否正確（非 NCC 核准碼 `CCA`、非 EN13319 標準號）
2. 若客戶有上傳購買憑證（發票），查看發票日期
3. 必要時聯繫客戶提供更多資訊

### 步驟三：手動覆蓋

判斷依據確認後：
1. 在工單詳情頁點擊「手動覆蓋保固」
2. 選擇正確的保固批次：
   - `legacy_2018_2022`：2018年1月 ～ 2022年10月生產，無標準保固
   - `v2_2022_2025`：2022年11月 ～ 2025年11月11日生產，保固 2 年
   - `v3_2025_onwards`：2025年11月12日以後生產，保固 1 年
3. 或直接勾選「在保固內」（`manualWarrantyOverride`）
4. **必填**：在備註說明覆蓋原因（如「客戶提供發票日期 2024/01/01，手動設為 v2 批次」）

### 步驟四：標記審核完成

覆蓋後在保固審核 Tab 標記「審核完成」，工單從待審核清單移除。

## 判斷依據優先順序

```
1. 序號解碼成功 → 用序號生產日期決定批次
   ↓（失敗時）
2. 手動批次覆蓋（manual_batch_override）→ 採用指定批次
   ↓（未設定時）
3. 客戶填寫的購買日期（warranty_date）→ 以購買日估算
   ↓（未提供時）
4. 無法判定 → 進入保固審核 Tab 人工處理
```

## 常見情況處理

### 序號是 NCC 核准碼（CCA 開頭）
客戶填錯了，請聯繫客戶重新提供序號：
> 回覆範例：「您好，您填寫的序號看起來是 NCC 核准碼（台灣電信認證號碼），而非產品序號。產品序號通常印在機身背面或包裝盒上，請確認後重新提供，感謝！」

### 序號看起來正確但解碼失敗
可能是型號不符（CR-4 和 CR-5L 序號位置不同）：
- 確認客戶填寫的型號是否正確
- 嘗試用另一型號的解碼規則

### 客戶有發票但序號顯示已過保
- 生產到銷售可能有 1-3 年的時間差（庫存品）
- 若客戶有明確購買日期憑證，可主管核准後以購買日計算保固
- 在備註詳細說明

## 注意事項

- 手動覆蓋需在備註留下明確依據，供未來查閱
- 若客戶提供偽造發票等情況，需上報主管，不得擅自決定
- 所有保固審核決定會影響後續是否免費維修，請謹慎評估
