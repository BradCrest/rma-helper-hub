## Phase 1｜自動保固判斷

**狀態**：📋 待開始  
**目標**：客戶提交申請後，系統根據序號與購買日期自動判斷保固是否有效，不需要人工查詢。

### 背景說明（AI 工具參考用）
目前流程：客服或技術人員在「故障登記」Tab 手動查詢序號對應的購買記錄並填寫保固判斷。  
目標：系統自動完成，技術人員只需確認或推翻 AI 的建議。

### 工作項目
- [ ] 確認現有序號資料庫結構（欄位、格式）
- [ ] 建立序號 → 購買日期 → 保固到期日的查詢 API
- [ ] 在「故障登記」頁面加入自動填入保固狀態（保固內/過保/無資料）
- [ ] 處理例外狀況：序號查無資料時的提示與人工接手流程
- [ ] 整合「銷貨匯入」功能（目前為 disabled，需啟用）

### 驗收標準
- [ ] 輸入任一有效序號，3 秒內自動顯示保固狀態
- [ ] 過保產品正確標示「已過保」且附上到期日
- [ ] 查無序號時顯示警示，不阻擋後續流程
- [ ] 測試案例：保固內 × 3、已過保 × 3、查無序號 × 2

### 技術備注
- 序號格式需與現有資料一致（確認是否有大小寫/前綴差異）
- 保固期限規則：需確認各產品型號保固年限是否不同
- 相依：「銷貨匯入」功能啟用（目前 disabled）

## Phase 2｜申請自動審核與通知

**狀態**：📋 待開始  
**目標**：客戶送出 RMA 申請後，系統自動審核資料完整性，並在 5 分鐘內發送確認信，不需要人工介入。

### 背景說明（AI 工具參考用）
目前流程：客服收到申請後，手動確認聯絡資訊、產品型號、問題描述是否填寫完整，再發送收件地址。  
目標：申請送出後全自動完成。

### 工作項目
- [ ] 定義「合格申請」的必填欄位清單
- [ ] 建立自動審核邏輯（缺資料 → 自動回信要求補件）
- [ ] 建立「申請確認信」Email 模板（含寄件地址、注意事項）
- [ ] 建立「補件要求信」Email 模板
- [ ] 設定觸發條件：申請送出後 → 自動執行審核 → 自動寄信
- [ ] 在 RMA 申請列表顯示「已自動通知」狀態標記

### 驗收標準
- [ ] 完整申請：送出後 5 分鐘內客戶收到確認信
- [ ] 不完整申請：客戶收到補件要求，並說明缺少哪個欄位
- [ ] 補件後重新送出，系統正確處理
- [ ] 測試案例：完整申請 × 5、缺聯絡電話 × 2、缺產品序號 × 2、缺問題描述 × 2

### 技術備注
- Email 發送需設定 SMTP 或串接 SendGrid / Resend 等服務
- 目前系統是否已有 Email 發送機制需確認
- 信件需支援繁體中文

### 背景說明（AI 工具參考用）
目前流程：收件後，客服手動更新狀態，手動發信通知客戶，技術人員手動從收件列表移到故障登記。  
目標：一個確認動作（收件登記）後，後續全部自動。

### 工作項目
- [ ] 在「收件處理」加入「確認收件」一鍵按鈕
- [ ] 確認收件後自動：更新狀態 → 寄送「已收到您的產品」通知信 → 自動推進到「故障登記」Tab
- [ ] 建立「收件通知信」Email 模板（含預估處理時間）
- [ ] 設定技術人員收件提醒（Email 或系統通知）
- [ ] 處理例外：收件時發現產品與申請不符的流程

### 驗收標準

- [ ] 案件自動出現在「故障登記」列表

- [ ] 測試案例 × 5

## Phase 4｜AI 故障分類輔助

**狀態**：📋 待開始  
**目標**：技術人員進入故障登記頁面時，AI 根據客戶描述預先填好故障類型和保固建議，技術人員只需確認或修改。

### 背景說明（AI 工具參考用）
目前流程：技術人員從頭填寫故障類型（無法開機/電池問題/充電問題等）和保固判斷，完全手動。  
這是整個流程中唯一必須保留人工的步驟（實體檢測），但填表時間可大幅縮短。

### 工作項目
- [ ] 建立故障類型分類模型（根據客戶問題描述 → 建議故障類型）
- [ ] 整合 RAG 知識庫：常見故障模式與處置方式
- [ ] 在故障登記頁面加入「AI 建議」區塊（預填，可修改）
- [ ] AI 建議信心度顯示（高/中/低）
- [ ] 建立故障-處置決策樹（保固內+特定故障 → 建議換新/送修/整新品）
- [ ] 記錄技術人員對 AI 建議的接受/修改率，持續優化

### 驗收標準
- [ ] AI 建議準確率 > 70%（前 100 筆案件）
- [ ] 頁面載入時 AI 建議在 3 秒內顯示
- [ ] 技術人員平均填表時間從 X 分鐘降至 Y 分鐘（需測量基準值）
- [ ] 測試案例：各故障類型至少 3 筆

### 技術備注
- 使用現有的 RAG 向量索引（已有 1462/1467 筆索引）
- 可能需要補充歷史案件資料作為訓練樣本
- 此 Phase 是技術難度最高的一個

## Phase 5｜自動客戶溝通引擎

**狀態**：📋 待開始  
**目標**：每個狀態轉換都自動觸發對應的客戶通知，並在客戶沒有回覆時自動跟進，不需要客服主動發信。

### 背景說明（AI 工具參考用）
目前流程：客服手動判斷該發什麼信，從 RMA 回覆介面手動操作。  
系統已有 AI 草稿功能，但觸發和發送仍需人工。

### 工作項目
- [ ] 建立狀態對應通知矩陣（哪個狀態變化 → 發哪封信）
- [ ] 建立完整 Email 模板庫（保固內換新/送修/整新品/過保報價等各情境）
- [ ] 設定自動觸發：故障登記完成 → 自動發「處理結果通知」
- [ ] 設定自動追蹤：客戶 3 天未回覆 → 自動發提醒信
- [ ] 設定自動追蹤：客戶 7 天未回覆 → 發第二封提醒 + 通知客服
- [ ] 支援多語言（繁中/英文）自動偵測
- [ ] 所有自動發送的信件記錄在案件歷史中

### 驗收標準
- [ ] 故障登記完成後，客戶在 10 分鐘內收到通知
- [ ] 3 天未回覆自動觸發提醒（誤差 < 1 小時）
- [ ] 信件正確對應案件資訊（姓名、產品、建議處置）
- [ ] 所有自動信件在系統中有記錄
- [ ] 測試案例：各情境至少 3 筆

### 技術備注
- 需確認現有 Email 發送功能的 API 限制
- 多語言判斷依據：客戶申請時的語言設定或 Email 網域

## 🔁 給 AI 工具的使用說明

如果你是 AI 工具（Claude、GPT、Cursor 等）正在接手這個計劃，請注意：

1. **每次開始工作前**，先讀這個檔案，了解哪個 Phase 是目前的焦點
2. **每次完成一個工作項目後**，更新對應的 checkbox（`- [ ]` 改為 `- [x]`）
3. **每次改變 Phase 狀態時**，更新頂部總覽表格的狀態欄
4. **測試完成後**，在「測試記錄」表格新增一行，記錄日期、測試項目、結果
5. **遇到決策點**（需要人工判斷的地方），在該工作項目下加一行 `⚠️ 待確認：[問題描述]`
6. **不要假設**系統的任何細節，先查 codebase 再動手
7. **系統 Repo**：https://github.com/BradCrest/rma-helper-hub
8. **本計劃檔案路徑**：`RMA_ZERO_HUMAN_PLAN.md`（repo 根目錄）
9. **交接給另一個 AI 時**，更新 `.ai_handoff/claude_review.md`（如果你是 Claude）或 `.ai_handoff/codex_review.md`（如果你是 Codex），記錄你做了什麼、有什麼問題。
